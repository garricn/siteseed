# siteseed

Automated data seeding for web apps. Discovers an app's data model (via OpenAPI spec, or UI form discovery as fallback), generates realistic data, and seeds it via API calls or form filling.

Primary use: seed test data before running sitetest e2e tests. Part of the site\* toolchain: sitecap (capture) → sitetest (test) → siteseed (seed) → sitefix (orchestrate).

## Tech Stack

- Node.js (ESM)
- Zod (operation schemas, API codegen)
- @faker-js/faker (data generation)
- js-yaml (seed plan files)
- sitecap/sitetest (optional, for UI discovery + form seeding)
- No build step

## CLI

```
siteseed discover --openapi <url>           Discover entities from OpenAPI spec
siteseed discover --url <url>               Discover entities from UI forms (via sitecap)
siteseed plan <seed-plan.yaml>              Show what will be seeded
siteseed run <seed-plan.yaml>               Execute seed plan
siteseed run --openapi <url> --auto         Discover + plan + seed in one step
```

## Architecture

Core logic (`lib/`) imports nothing from API layer (`bin/`, `generated/`).
Operations are the bridge — they import core logic and export handlers consumed
by generated surfaces. All API surfaces (MCP, REST, OpenAPI, function-calling
tools) are generated from `lib/operations.js` via `scripts/generate.js`.

### Two discovery modes

**API mode (preferred)** — reads OpenAPI/Swagger spec, extracts schemas, infers relationships from field names (`customerId` → `customers`), builds dependency graph.

**UI mode (fallback)** — uses sitecap accessibility tree to discover forms, infers field types from input labels/types, seeds by filling forms via sitetest.

### Entity Graph

Core data structure — directed acyclic graph of entities with typed fields and dependency edges. Topological sort determines seed order.

### Seed Plans

YAML files describing what to seed: entity names, counts, field overrides. Editable by humans, generated by `siteseed discover`.
