# SEED-2: CLI

## Context

`bin/siteseed.js` is a stub. All core modules are implemented: discover (DSC-1/2), generate (GEN-1/2/3), seed (SEED-1). The CLI wires them together as the primary user interface.

CLAUDE.md defines the CLI interface:
```
siteseed discover --openapi <url>           Discover entities from OpenAPI spec
siteseed discover --url <url>               Discover entities from UI forms (Phase 2)
siteseed plan <seed-plan.yaml>              Show what will be seeded
siteseed run <seed-plan.yaml>               Execute seed plan
siteseed run --openapi <url> --auto         Discover + plan + seed in one step
```

## Goals

- Implement three subcommands: `discover`, `plan`, `run`
- Parse CLI args with `node:util parseArgs` (built-in, no dependency)
- Output structured JSON for programmatic consumption, human-readable summaries to stderr
- `--auto` mode on `run`: discover → plan → seed in one step
- Exit codes: 0 success, 1 errors occurred (partial), 2 fatal

## Subcommands

### `siteseed discover --openapi <url|file> [--base-url <url>]`

- Call `discoverFromOpenAPI(openapi, baseUrl)`
- Print entities as JSON to stdout
- Exit 0

### `siteseed plan --openapi <url|file> [--count <n>] [--out <file>]`

- Call `discoverFromOpenAPI(openapi)` then `generatePlan(entities, { baseUrl, count })`
- If `--out`: write YAML to file via `writePlan(plan, filePath)`
- Print YAML to stdout
- Exit 0

### `siteseed run <plan-file> --openapi <url|file> [--auth <header>] [--dry-run]`

- Read plan from positional arg or stdin
- Call `executeSeed(planSource, { openapi, auth, dryRun, faker })`
- Print result JSON to stdout (seeded counts + errors)
- Exit 0 if no errors, 1 if errors present

### `siteseed run --openapi <url|file> --auto [--count <n>] [--auth <header>] [--dry-run]`

- Discover → plan → seed in one step
- Equivalent to: discover + plan + run pipeline
- Call `discoverFromOpenAPI`, `generatePlan`, `executeSeed` with generated plan YAML
- Print result JSON to stdout
- Exit 0/1

## Argument Parsing

Use `node:util parseArgs`:

```
Options:
  --openapi <string>    OpenAPI spec URL or file path
  --base-url <string>   Override base URL
  --count <string>      Records per entity (parsed to number, default 5)
  --auth <string>       Authorization header value
  --out <string>        Output file path (plan command)
  --dry-run             Preview without executing
  --auto                Auto mode: discover + plan + seed
  --help                Show usage
```

Positional args: `args.positionals[0]` = subcommand, `args.positionals[1]` = plan file (for `run`)

### Subcommand routing

| positionals[0] | behavior |
|---|---|
| `discover` | discover subcommand |
| `plan` | plan subcommand |
| `run` | run subcommand |
| undefined / `--help` | print usage, exit 0 |
| anything else | print usage, exit 2 |

## Phases

### Phase 1: Arg parsing and routing (`bin/siteseed.js`)

- `parseArgs` with options defined above
- Route to handler based on subcommand
- `--help` or no args → print usage to stderr, exit 0
- Unknown subcommand → print usage to stderr, exit 2

### Phase 2: Subcommand handlers (`bin/siteseed.js`)

All handlers in the same file (thin shell, no complex logic).

- `handleDiscover(args)`: validate `--openapi` required, call `discoverFromOpenAPI`, JSON.stringify to stdout
- `handlePlan(args)`: validate `--openapi` required, discover + generatePlan + writePlan, output YAML
- `handleRun(args)`: two modes based on `--auto`:
  - With `--auto`: discover → plan → executeSeed
  - Without `--auto`: read plan file from positional, call executeSeed
  - Validate: `--openapi` required in both modes; plan file required without `--auto`

### Phase 3: Error handling

- Wrap each handler in try/catch
- Caught errors → print message to stderr, exit 2
- Seed errors (partial failure) → print result with errors to stdout, exit 1

### Phase 4: Tests

Create `test/cli.test.js` using `execSync`/`execFile` to run the CLI:

- **No args prints usage**: run `node bin/siteseed.js` → assert stderr includes "usage" or "siteseed", exit 0
- **Unknown subcommand exits 2**: `node bin/siteseed.js badcmd` → assert exit code 2
- **Discover outputs JSON**: `node bin/siteseed.js discover --openapi test/fixtures/petstore.json` → assert stdout parses as JSON with `entities` array
- **Plan outputs YAML**: `node bin/siteseed.js plan --openapi test/fixtures/petstore.json` → assert stdout includes `version: 1`
- **Plan with --count**: `--count 3` → assert plan entities have count 3
- **Plan with --out writes file**: `--out /tmp/test-plan.yaml` → assert file exists, clean up
- **Run --auto --dry-run**: `node bin/siteseed.js run --openapi test/fixtures/petstore.json --auto --dry-run` → assert stdout parses as JSON with `seeded` object
- **Run without --openapi exits 2**: `node bin/siteseed.js run --auto` → assert exit code 2
- **Help flag**: `node bin/siteseed.js --help` → assert exit 0, stderr includes usage

## Design Decisions

1. **`node:util parseArgs`** — built-in since Node 18.3. No external dependency. Sufficient for our option set.

2. **JSON to stdout, messages to stderr** — follows Unix convention. Allows piping output to other tools. Human-readable summaries (entity count, error count) go to stderr.

3. **Single file** — the CLI is a thin shell over library functions. No need for separate command modules — each handler is ~10 lines calling existing library exports.

4. **`--count` as string parsed to number** — `parseArgs` doesn't support number types. Parse with `Number()` after extraction.

5. **`--auto` combines all steps** — the most common usage pattern. Without it, users need to run discover → plan → run separately, which is useful for customizing the plan YAML between steps.
